## Warnings

### WR-01: Repeated-failure alert message body is hardcoded to `weekly_metrics` but fires for any `partial`-returning task

**File:** `scripts/sports_system_runner.py:7446-7447`

**Issue:** The `if result.get("status") == "partial":` block in `main()` fires unconditionally on any task that returns `status="partial"`. The alert body on line 7447 is hardcoded as `"Weekly metrics data is degraded — check run log"`. Currently only `weekly_metrics_task()` emits `partial`, so this is silent. However, `sync_slip_bankroll` can set `status="partial"` in `rebuild_slip_bankroll`, and it is plausible that future maintenance adds another partial-returning task routed through `run_task()`. When that happens, the Telegram alert will describe "Weekly metrics data is degraded" for an unrelated task failure, causing an operator to investigate the wrong component under a live incident.

**Fix:** Make the message body task-aware:
```python
if result.get("status") == "partial":
    _obs03_streak = trailing_failure_streak(args.task) + 1
    if _obs03_streak >= REPEATED_FAILURE_THRESHOLD:
        send_telegram(
            f"🔁 REPEATED FAILURE: {args.task} partial {_obs03_streak} times in a row\n"
            f"Task returned degraded result — check run log"
        )
```

### WR-02: `test_low_prob_or_negative_ev_gets_zero` only exercises the zero-floor gate, not the EV gate it claims to test

**File:** `scripts/test_build_slips.py:259-276`

**Issue:** The test name and docstring both state "Sub-0.58-prob **or** EV<=0 slip gets stake_units == 0.0". The test fixture sets `combined_probability=0.55, combined_ev_score=0.50` — the zero-floor gate fires (`0.55 < 0.58`) before the EV gate is even evaluated. The EV gate path (`combined_ev_score <= 0` with a probability that would otherwise qualify for staking) is completely absent from this test class. Because `build_slips.main()` constructs staked slips from `apply_confidence_stakes`, a slip with `combined_probability=0.80` and `combined_ev_score=0.0` would reach `confidence_stake()` and should return `0.0`, but this integration path is not verified via the `main()` call chain. If `stake_sizing.confidence_stake` or `apply_confidence_stakes` were ever changed, the integration regression would go undetected.

**Fix:** Add a complementary test case covering the EV gate integration:
```python
def test_zero_ev_gets_zero_stake(self) -> None:
    """EV gate: combined_ev_score <= 0 -> stake_units == 0.0 even when prob is high."""
    bankroll = 100.0
    # combined_probability=0.80 would normally give 2.5%, but ev_score=0.0 kills it
    self._fixture_payload = self._make_fixture_payload(
        combined_probability=0.80, combined_ev_score=0.0
    )
    with tempfile.TemporaryDirectory() as tmp:
        tmp_path = Path(tmp)
        pnl_dir = tmp_path / "data" / "pnl"
        pnl_dir.mkdir(parents=True, exist_ok=True)
        (pnl_dir / "bankroll.json").write_text(
            json.dumps({"current_bankroll": bankroll}), encoding="utf-8"
        )
        result = self._run_main_with_root(tmp_path)
    slips = result["slips"]["safest_2_leg"]
    self.assertEqual(slips[0]["stake_units"], 0.0)
```

## Info

### IN-01: Calibration dedup — `generate_projections.py` now calls `load_calibration_factor(sport)` with no `path=` override, using the module-level `CALIBRATION_PATH` from `calibration.py`

**File:** `scripts/generate_projections.py:398`

**Issue:** The deleted local `load_calibration_factor` in `generate_projections.py` resolved the file path using `DATA / "research" / "calibration.json"` where `DATA = Path.home() / "sports_picks" / "data"`. The imported version from `calibration.py` uses `ROOT / "data" / "research" / "calibration.json"` where `ROOT = Path(__file__).resolve().parents[1]` — i.e., the repo root resolved from `calibration.py`'s actual location. On the installed Hermes machine these resolve to the same file. However, the two derivation strategies differ: the old one was home-dir-anchored (`~/sports_picks/`), the new one is filesystem-anchored relative to the module file. If the repo were ever symlinked or installed at a non-home path, the resolved paths could diverge silently. This is a latent environment assumption, not a bug in the current deployment.

**Fix:** No action required for the current deployment. Document the assumption in `calibration.py` that `ROOT` must resolve to the same directory as the runner's `BASE = Path.home() / "sports_picks"`.

### IN-02: No integration test covers the `main()` repeated-failure alert path for `partial` status

**File:** `scripts/test_weekly_metrics.py:584-665`

**Issue:** `TestPartialVisibility.test_single_partial_no_hard_fail` verifies the sub-component (`trailing_failure_streak` returns 1 for a single prior partial) but does not call `main()` to confirm that a single partial actually returns exit code 0 and sends no `🔁 REPEATED FAILURE` Telegram. The component-level evidence is sound, but the integration contract (exit code 0, no alert, `dispatch_alerts` not called with status!=ok) is untested at the `main()` boundary. A future refactor of the `try/except` structure in `main()` could break the single-partial safety guarantee without failing any existing test.

**Fix:** Add an integration-level test that calls `main()` via `subprocess` or mocks the full `main()` flow with a synthetic `run_task` that returns `{"status": "partial"}` and asserts `send_telegram` is called exactly once (the DEGRADED digest), not twice (no repeated-failure alert).
