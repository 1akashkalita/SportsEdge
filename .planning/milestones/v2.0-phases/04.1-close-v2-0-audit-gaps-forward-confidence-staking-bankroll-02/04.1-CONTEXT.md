### Forward confidence staking (BANKROLL-02)
- **D-01:** Forward staking is **ON by default, no feature flag** — consistent
  with P4 **D-09** (auto-apply, no flag) and the milestone's "stop babysitting"
  goal. The stake is already hard-bounded (≤2.5% of bankroll per slip, zero-floor
  below 0.58 prob / EV≤0), so bounds + tests are the safety net, not a flag. Wire
  the existing `stake_sizing.apply_confidence_stakes(slips, start_of_day_bankroll)`
  into the daily slip-build path so `build_slips.py` stops emitting the flat
  `stake_units: 1.0` placeholder.
- **D-02:** Forward stakes size off the **live `bankroll.json` `current_bankroll`
  read at slip-build time** — this is the de-facto "start-of-day bankroll
  snapshot" (P3 **D-14**), because today's games haven't graded yet so the value
  equals prior-day close. NBA and MLB build their slips pre-game on the same
  calendar day, so both sports naturally share the same snapshot (honors D-14's
  "same start-of-day snapshot, no intra-day compounding").
- **D-03:** **Accept lag.** If grading is behind (yesterday's slips not yet
  graded when today's build runs), `bankroll.json` lags and a forward stake may
  differ slightly from what a later rebuild computes. That is acceptable —
  `rebuild_bankroll` remains the **authoritative re-stating** (P3 **D-12**), and
  forward staking is best-effort against the bankroll known at build time.
- **D-04:** **Missing-snapshot fallback** (Claude's-discretion default, money-safe):
  if `bankroll.json` is missing/unreadable, record `stake_units = 1.0` and log a
  warning — never 0, never crash; a later rebuild corrects it. Matches the
  system's "missing data → defensive, never raise" contract.
- **D-05:** **No grading change.** `grade_slips.py:320` already reads
  `slip.get("stake_units") or 1.0` verbatim — once build-time stakes are real,
  grading consumes them with no edit. The stake simply stops being a placeholder.

### Daily Prop-Accuracy refresh (BANKROLL-04 / METRICS-01 freshness)
- **D-06:** Refresh `Prop Accuracy` on the **daily grade/sync path**
  (`grade_slips` → `sync_slip_bankroll`), which already loads and `save_workbook_atomic`s
  `master_pnl.xlsx` under its write lock — colocate `refresh_prop_accuracy(wb)`
  before the save, mirroring the existing `rebuild_bankroll` pattern at
  `runner:5681`. The sheet stays live all week at **zero extra lock cost**, and
  **`weekly_metrics` stays read-only** on `master_pnl.xlsx` (its current contract
  — writes only `calibration.json`). Do NOT flip `weekly_metrics` to read-write.

### Claude's Discretion
- **D-07 (WR-03 dead-loop visibility):** A persistent `weekly_metrics status=partial`
  must never read green. Two-part fix: (a) mark the **Telegram digest visibly
  "⚠ degraded"** whenever `status=partial`, every run, so a degraded cycle is
  immediately visible; and (b) feed **persistent** partials into the existing
  failure-streak / health signal (OBS-03) so repeated partials raise an alert. Do
  **NOT** hard-fail the task (non-zero exit) on a single transient partial — that
  would stop the digest delivering and add cron noise. The planner confirms how
  OBS-03 detects failures (exit code vs run-log/health parse) and wires the
  persistent-partial signal accordingly.
- **D-08 (calibration dedup):** `calibration.py` becomes the **canonical** owner
  of `load_calibration_factor` (it's the richer copy — has the `path` param and
  owns the calibration domain). `generate_projections.py` imports it
  (`from calibration import load_calibration_factor`) and deletes its local copy
  (`generate_projections.py:277-293`). Behavior must stay **byte-identical**
  (same `CALIBRATION_PATH`, same `[0.85, 1.20]` clamp, same default 1.0). A test
  asserts both call sites resolve the same factor. `generate_projections.py` runs
  as a subprocess from `scripts/` and `calibration.py` is in `scripts/`, so the
  import is on `sys.path` — confirm no circular import. Preserves P4 **D-13**
  (no projection/gate behavior change).
- Module placement, exact task/function names, and test-file names are the
  planner's discretion, subject to the contracts above.
</decisions>

<specifics>
## Specific Ideas

- **Forward stake = same formula as rebuild** (so forward ≈ rebuild in the
  normal case), from `stake_sizing.confidence_stake` (P3 D-03/D-04/D-05):
  ```
  if combined_ev_score <= 0:          stake = 0.0
  elif combined_probability < 0.58:   stake = 0.0
  elif combined_probability >= 0.75:  stake = 0.025 * start_of_day_bankroll
  elif combined_probability >= 0.65:  stake = 0.015 * start_of_day_bankroll
  else:  # 0.58 <= prob < 0.65        stake = 0.0075 * start_of_day_bankroll
  # start_of_day_bankroll = bankroll.json current_bankroll at build time (D-02)
  ```
- **Verification anchors (success-aligned):**
  1. **BANKROLL-02 live:** a daily-build test asserts a high-prob (+EV) slip gets
     `2.5% × current_bankroll`, a `<0.58`/`EV≤0` slip gets `0`, and **no** slip is
     emitted with the flat `1.0` placeholder when `bankroll.json` is present.
  2. **Snapshot:** with `bankroll.json` missing/unreadable, the build falls back
     to `stake_units=1.0` and logs (D-04) — does not crash, does not emit 0.
  3. **Prop-Accuracy freshness:** after a daily grade run, `Prop Accuracy`
     reflects the just-graded results; `weekly_metrics` opens `master_pnl.xlsx`
     read-only (no write lock acquired).
  4. **WR-03:** a forced `weekly_metrics` partial produces a visibly-degraded
     Telegram digest **and** trips the failure-streak/health signal on repeated
     partials; a single transient partial does not hard-fail the task.
  5. **Calibration dedup:** `load_calibration_factor` resolves identically via
     `calibration.py` and the now-importing `generate_projections.py`; the P4 D-13
     integrity test (no verdict/gate change) still passes.
- Every task touched must re-confirm it stays under the **660s** budget.
</specifics>
