## Summary

Phase 04.1 is a **pure wiring/hardening pass** — every formula, bound, and integrity guarantee is already built and tested. The four gaps identified by the v2.0 milestone audit are: (1) `build_slips.py` hardcodes `stake_units=1.0` instead of calling `stake_sizing.apply_confidence_stakes`; (2) `refresh_prop_accuracy` is only called inside `rebuild_slip_bankroll`, not on the daily `sync_slip_bankroll` path; (3) `load_calibration_factor` has an independent duplicate in `generate_projections.py` instead of importing from `calibration.py`; (4) `weekly_metrics_task` returns normally (exit 0) on `status=partial` and the OBS-03 streak mechanism never sees it as a failure.

All four fixes are localized: gap 1 is one call site in `build_slips.py:main()`; gap 2 is one line added to `sync_slip_bankroll` before its `save_workbook_atomic`; gap 3 is a one-line import substitution plus deletion of 17 lines in `generate_projections.py`; gap 4 is a conditional `append_run_record` in `weekly_metrics_task` plus a degraded marker in the Telegram digest string. No schema changes, no gate changes, no new tasks.

The critical money-safety constraint is gap 1: `build_slips.py` is invoked as a subprocess by `grade_slips.ensure_slip_defs`, which is called from `grade_slips.grade_slips_for_date`, which is called from `_grade_slips_then_sync`. The staked slips are written to `data/research/slips/slips_<date>.json` by `build_slips.write_outputs`, and `grade_slips` reads `stake_units` verbatim from that file — so the fix must apply `apply_confidence_stakes` to the full slip list inside `build_slips.main()` before calling `write_outputs`.

**Primary recommendation:** Wire all four fixes in a single focused plan with one task per gap; treat the forward-staking wiring as the highest-risk item (real-money path) and verify it first with the existing `test_stake_sizing.py` + a new daily-build fixture test.

### Pattern 1: Forward Staking Wiring (Gap 1)

**What:** `build_slips.py:main()` currently writes `stake_units: 1.0` via `make_slip()`. The fix applies `apply_confidence_stakes` to the assembled slip list BEFORE calling `write_outputs`, reading `bankroll.json` directly.

**Exact wiring point:** `build_slips.py:main()` after `payload = build_slips(...)` and before `write_outputs(payload, date)`. The `payload["slips"]` dict maps category→list-of-slips; the staking must be applied per-slip across all categories.

**Bankroll read:** Inside `build_slips.main()`, read `ROOT / "data" / "pnl" / "bankroll.json"` (where `ROOT = Path(__file__).resolve().parents[1]`). Extract `current_bankroll`. If the file is absent or unreadable, use `1.0` as fallback and print a warning (D-04). Never crash, never return 0.

**Key pitfall (money-safety):** The `slips` value in `payload["slips"]` is a `dict[str, list[dict]]` (category → list), NOT a flat list. `apply_confidence_stakes` expects a flat list. The fix must flatten, apply stakes, then re-partition by category, OR call `apply_confidence_stakes` on each category list separately and reassemble. Either approach is correct; the per-category loop is simpler.

```python
# Source: scripts/stake_sizing.py:71 (apply_confidence_stakes) + scripts/build_slips.py:630 (main)
# D-04: bankroll read with fallback
bankroll_path = ROOT / "data" / "pnl" / "bankroll.json"
start_of_day_bankroll = 1.0  # fallback — means stake = tier% * 1.0 (still non-zero for +EV slips)
try:
    if bankroll_path.exists():
        bk = json.loads(bankroll_path.read_text(encoding="utf-8"))
        raw = float(bk.get("current_bankroll") or 0)
        if raw > 0:
            start_of_day_bankroll = raw
        else:
            print("[build_slips] WARNING: bankroll.json current_bankroll <= 0; using fallback 1.0")
    else:
        print("[build_slips] WARNING: bankroll.json not found; using fallback stake_units=1.0")
except Exception as exc:
    print(f"[build_slips] WARNING: bankroll.json read failed ({exc}); using fallback stake_units=1.0")

# Apply stakes per category (D-01: all 6 categories eligible)
from stake_sizing import apply_confidence_stakes
staked_slips: dict[str, list] = {}
for category, slip_list in payload["slips"].items():
    staked_slips[category] = apply_confidence_stakes(slip_list, start_of_day_bankroll)
payload["slips"] = staked_slips
```

**Why `stake_units: 1.0` fallback is money-safe:** `apply_confidence_stakes` with `start_of_day_bankroll=1.0` still applies the formula — a high-probability slip gets `0.025 * 1.0 = 0.025`. This is an extremely small dollar amount, but it is NOT a flat-1.0 placeholder. The real money-safety comes from D-04 being a fallback for exceptional circumstances (file missing/corrupt), not the normal case. The CONTEXT.md wording "record `stake_units = 1.0`" in D-04 means the raw fallback value, not the formula result — so the fallback should write the literal `1.0` when the bankroll read fails entirely, to match the prior placeholder behavior exactly. See Anti-Patterns below.

**Correction to the above:** Re-reading D-04: "if `bankroll.json` is missing/unreadable, record `stake_units = 1.0` and log a warning." This means the fallback should write `stake_units=1.0` directly to every slip when the bankroll read fails, rather than calling `apply_confidence_stakes(slips, 1.0)` which would produce fractional values like `0.025`. The cleanest implementation: on read failure, skip calling `apply_confidence_stakes` and leave `stake_units` at its existing `make_slip` value of `1.0`.

### Pattern 4: WR-03 Partial Visibility (Gap 4)

**What:** When `weekly_metrics_task` finishes with `result["status"] == "partial"`, two things must happen: (a) the Telegram digest gets a "⚠ degraded" prefix, and (b) a run-log record with `status="partial"` is appended to `run_log.jsonl` so that `trailing_failure_streak("weekly_metrics")` counts it toward the OBS-03 repeated-failure threshold.

**OBS-03 mechanism (VERIFIED):** `trailing_failure_streak(task)` reads `run_log.jsonl`, filters records to the given task, walks backward counting consecutive records where `status in {"error", "timeout"}`, stops at `status == "ok"`. It does NOT currently count `status="partial"`. After this fix, it should also count `"partial"` — change the check to `status in {"error", "timeout", "partial"}`.

**Current flow:** `weekly_metrics_task` sets `result["status"] = "partial"` at lines 7309 and 7331 but returns normally. `main()` calls `dispatch_alerts(task, result)` — `dispatch_alerts` at line 1251 checks `result.get("status") != "ok"` and returns early if non-ok, so the Telegram digest is NOT sent by `dispatch_alerts`. The task's own Telegram call at line 7339 (`send_telegram(digest)`) runs regardless of `status`, but the digest string contains no "degraded" marker. The `finally` block at line 7497 calls `append_run_record` — but `_run_status` is still `"ok"` because the `except Exception` branch in `main()` was never entered.

**Fix mechanism — two sub-parts:**

Sub-part A (Telegram degraded marker): At the end of `weekly_metrics_task`, after all delivery attempts, check `result["status"]` and modify the digest string. Cleanest: add a prefix to the digest BEFORE calling `send_telegram`. Already inside the task, so the digest variable is in scope. Add at the beginning of section 3:
```python
# D-07 WR-03: mark digest visibly degraded when status=partial
if result.get("status") == "partial":
    digest = "⚠️ WEEKLY METRICS DEGRADED (partial data)\n\n" + digest
```

Sub-part B (run_log.jsonl record): `weekly_metrics_task` needs to emit a `status="partial"` record to `run_log.jsonl` so `trailing_failure_streak` sees it. The cleanest approach: call `append_run_record` directly inside `weekly_metrics_task` when `result["status"] == "partial"`, using the same structure that `main()`'s `finally` block uses. This writes the JSONL entry from within the task function, BEFORE `main()`'s `finally` block also writes one. The `finally` block will write a second record with `status="ok"` (because `_run_status` is never changed by `weekly_metrics_task`). To prevent double-counting, the simplest fix is to modify `trailing_failure_streak` to count `"partial"` as a failure AND modify `main()`'s `finally` JSONL write to use `result.get("status", "ok")` for the `weekly_metrics` task when the task returns a `status="partial"` result, OR expose `_task_result` to the `finally` block. The existing `_task_result = result` assignment at line 7432 already captures the result — so `main()`'s `finally` block can check `_task_result.get("status")` and write `"partial"` instead of `"ok"` to the JSONL record.

**Recommended approach for Sub-part B:** Modify `main()`'s `finally` block to use `_task_result.get("status", "ok")` as the JSONL status when `_task_result is not None` and the task did not raise an exception. This requires one-line change in `finally`:
```python
# In the finally block, replace:
"status": _run_status,
# With:
"status": (_task_result.get("status", "ok") if _task_result is not None else _run_status),
```
This cleanly propagates `status=partial` to the JSONL without adding duplicate writes. Then update `trailing_failure_streak` to count `"partial"` as a failure:
```python
if status in {"error", "timeout", "partial"}:
    count += 1
```

**Single-transient-partial safety (D-07):** `REPEATED_FAILURE_THRESHOLD` defaults to 2. A single partial produces `trailing_failure_streak = 1` after the record is written — the streak check `_obs03_streak >= 2` fails, no repeated-failure alert fires. Only two consecutive partials trigger the alert. This matches D-07 ("Do NOT hard-fail on a single transient partial").

**Note:** `dispatch_alerts` checks `result.get("status") != "ok"` and skips alert delivery for non-ok results. This is fine — `weekly_metrics_task` sends its own Telegram digest directly (line 7339), bypassing `dispatch_alerts`. The degraded marker approach works regardless of this.

### Anti-Patterns to Avoid

- **Calling `apply_confidence_stakes(slips, 1.0)` on bankroll-read failure:** This produces fractional stakes like `0.025` instead of the flat `1.0` placeholder specified in D-04. On read failure, skip the staking call entirely and leave `stake_units=1.0` as set by `make_slip`.
- **Applying `apply_confidence_stakes` to a flat list of all slips then rebuilding the dict:** The `payload["slips"]` structure is `dict[str, list]` (category → slips). Flattening loses category membership. Apply per-category to preserve structure.
- **Adding `refresh_prop_accuracy` OUTSIDE the `if not dry_run:` block:** `dry_run=True` is used in tests. Calling `refresh_prop_accuracy` in dry_run mode would mutate the in-memory workbook even when the caller does not expect writes. Keep it inside `if not dry_run:`.
- **Modifying `dispatch_alerts` to handle `status=partial`:** The existing pattern is fine; the Telegram digest is sent by the task itself, not by `dispatch_alerts`.
- **Using `__import__("calibration")` lazy import inside `generate_projections.py`:** The module is already a subprocess called from `scripts/`, so a top-level `from calibration import load_calibration_factor` is both clean and consistent with existing top-level imports (`from line_timing import ...`, `from workbook_io import ...`).
- **Adding `weekly_metrics` to `dispatch_alerts` handling:** `weekly_metrics` sends its own digest. Adding it to `dispatch_alerts` would send two Telegram messages.

## Common Pitfalls

### Pitfall 1: build_slips is invoked as a subprocess — stakes must be in the JSON file

**What goes wrong:** If the stake fix is applied in `sports_system_runner.py` (e.g., after reading the JSON file) instead of inside `build_slips.py`, then `grade_slips.ensure_slip_defs` will invoke `build_slips.py` on days the JSON is missing and write the file with `stake_units=1.0`. The JSON file IS the contract between build and grade.

**Why it happens:** `build_slips.py` is a standalone subprocess. `grade_slips.ensure_slip_defs` shells out to it. The runner never post-processes the slips JSON.

**How to avoid:** The fix MUST be inside `build_slips.main()`, before `write_outputs`. Verify: check that `data/research/slips/slips_<date>.json` contains real (non-1.0) `stake_units` values after the fix.

**Warning signs:** `slips_<date>.json` shows `"stake_units": 1.0` on every slip despite the fix being "applied." Check whether the fix is in `main()` (correct) or in a separate post-processing step that only runs in one code path.

### Pitfall 2: `payload["slips"]` is a dict of lists, not a flat list

**What goes wrong:** Calling `apply_confidence_stakes(payload["slips"], bankroll)` passes a dict instead of a list — the function will silently iterate over dict keys (strings) and return a list of dicts with those string keys as values, corrupting the output.

**Why it happens:** `build_slips` returns `{"slips": {"safest_2_leg": [...], "highest_ev": [...], ...}}`.

**How to avoid:** Iterate `payload["slips"].items()` and apply stakes per category.

### Pitfall 3: `sync_slip_bankroll` dry_run guard

**What goes wrong:** `refresh_prop_accuracy(wb)` added outside the `if not dry_run:` block runs in test fixtures and mutates the in-memory workbook in tests that expect the workbook to be unmodified.

**Why it happens:** `sync_slip_bankroll` has a `dry_run` parameter used in testing.

**How to avoid:** Place `refresh_prop_accuracy(wb)` inside `if not dry_run:`, before `save_workbook_atomic`. The pattern at `rebuild_slip_bankroll:5681` already does this correctly — mirror it exactly.

### Pitfall 4: Calibration import placement vs call-time semantics

**What goes wrong:** `generate_projections.py:416` (`cal_factor = load_calibration_factor(sport)`) is called INSIDE `build_projection()` at projection time, not at import time. If the import statement inadvertently introduces an import-time read of `calibration.json`, calibration would be cached at module load instead of re-read per projection call.

**Why it happens:** Misplacing side-effectful code in module scope.

**How to avoid:** `from calibration import load_calibration_factor` imports only the function object — the function is called later by `build_projection()`, so it reads `calibration.json` at call time exactly as before. No behavioral change. Confirmed: `calibration.py` has no module-level I/O (its `CALIBRATION_PATH` is just a `Path` object construction).

### Pitfall 5: Double JSONL record for weekly_metrics

**What goes wrong:** If `append_run_record` is called inside `weekly_metrics_task` AND `main()`'s `finally` block also writes a record, two records appear for the same run. `trailing_failure_streak` counts both — after one partial run, streak=2 (both records count), falsely triggering the repeated-failure alert on a single transient partial.

**Why it happens:** Trying to write the partial status from inside the task without adjusting the `finally` block.

**How to avoid:** Use the recommended approach: modify `main()`'s `finally` block to propagate `_task_result.get("status")` to the JSONL record when the task returned a result. No record is written inside `weekly_metrics_task`.

### Pitfall 6: Zero stake when bankroll.json missing

**What goes wrong:** Calling `apply_confidence_stakes(slips, start_of_day_bankroll=0.0)` when the bankroll file is missing or corrupted produces `stake_units=0.0` for all slips (because `0.025 * 0 = 0`). A zero stake is silently un-bet — the operator gets no alert and the slip appears staked but is actually dead.

**Why it happens:** Forgetting that the formula multiplies by bankroll. Bankroll=0 is technically possible if `current_bankroll` is 0 (deep drawdown).

**How to avoid:** The read block must check `raw > 0` before using the bankroll value; fallback to `1.0` (not `0.0`) on any edge case (missing file, corrupt JSON, zero balance). See Pattern 1 code example.
