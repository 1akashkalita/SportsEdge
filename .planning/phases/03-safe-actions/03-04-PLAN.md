<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workbook-derived slip values / operator note → rendered HTML | Slip fields and the operator's own note text are rendered into the page; if not escaped they could inject script (stored XSS on a local tool). |
| browser form → POST routes | The forms submit `task`/`date`/`slip_id`/`note`; server-side validation (whitelist, write helper) is the authority — the template is only the UI surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09 | Tampering (XSS) | slip values + Operator Note rendered in slips.html | mitigate | Jinja autoescaping stays ON; the Phase 2 no-safe-filter rule is preserved for all workbook-derived data and the note text — `{{ value }}` is HTML-escaped by default. |
| T-03-10 | Tampering | refresh task value from the dropdown | mitigate | The dropdown only offers the five `ALLOWED_TASKS`; server-side `ALLOWED_TASKS` whitelist in the route (Plan 03 T-03-05) is the real enforcement — the dropdown is convenience, not the control. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No packages installed this phase (03-RESEARCH "Package Legitimacy Audit"); Pico.css / Chart.js are existing CDN tags from Phases 1-2, unchanged here. |

Auth / CSRF: accepted / out-of-scope — single-operator `127.0.0.1`-only tool (REQUIREMENTS.md Out of Scope; design §8).
</threat_model>
